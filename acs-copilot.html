The question

Call Summary put AI after the call. Copilot in Rooms asked a harder question: what does AI presence look like during the call itself? Not as post-processing, not as a parallel chat window — as an active, responsive assistant inside the calling composite while the call is live.

The spatial problem was interesting. Calling UIs already carry a lot: video gallery, control bar, captions, participants panel. Adding an AI layer to that environment required a model that was present when needed, invisible when not, and couldn't disrupt the primary experience of the call. I designed a Copilot side pane with a toggle model — opt-in, collapsible, living at the same level as the participants panel — and scoped it explicitly to the Sample Builder as a validation surface before committing it to the production component API.

Design decisions

01 The Spatial
Question

Where does AI live inside a calling interface?

I considered four spatial models before landing on the side pane. An overlay floats on top of the video gallery — it blocks participants and competes with the primary content of the call. A bottom band pushes the gallery up and reflows the layout — it changes the experience for everyone in the call, including people who haven't opted in to AI assistance. A floating panel is flexible but creates a persistent element with no stable relationship to the rest of the UI. A side pane slots into the same spatial category as the participants panel and captions — a panel that opens from the edge, takes a predictable portion of the layout, and closes without affecting what remains.

The side pane model also had a practical advantage: the OPS/contact-center workstream had already established a side pane pattern for supplementary call surfaces (dialpad, OBO selection, supervisor controls). Copilot fits that category — it's a contextual surface, not a primary control. Reusing the pattern reduces cognitive overhead for developers building on the library and for end users navigating the call.

02 Ambient,
Not Default

Opt-in with a toggle, collapsible without ending the session

Copilot shouldn't be on by default for a call. Not every call needs it, and presenting AI assistance as a persistent element of the calling UI changes the default character of the product — it becomes an AI product, not a calling product. The toggle model keeps Copilot ambient: available but not present unless explicitly opened.

The toggle design

A Copilot button in the call control bar opens the side pane. The pane is collapsible — closing it returns the gallery to full width without a state transition that feels disruptive. When the pane is closed but Copilot has been activated in the call, I designed a small persistent indicator in the control bar to confirm the session is still active. The pane can be reopened from that indicator at any point. The call continues regardless of whether the pane is open or closed.

03 Sample Builder
as Validation

Don't commit to the API until you've proven the pattern

The spec explicitly scoped Copilot in Rooms as a Sample Builder experimentation surface — not full production support in the UI Library composites. This was a design strategy decision, not a product-resource hedge. Shipping something into the CallComposite API means it's part of the public contract: breaking it is a version bump, changing it requires migration guidance, removing it requires deprecation notices.

The Sample Builder has a much lower commitment surface. Developers using it can see and deploy the Copilot experience without the library itself committing to the API shape. If the interaction model doesn't work in practice — if the pane placement is wrong for mobile, if the toggle is confusing, if Copilot's response cadence doesn't fit the calling context — those lessons can be learned and corrected before the pattern is frozen into a production API.

Why this is the right approach for exploratory AI UX

Call Summary established the same pattern: ship in the Sample Builder first, prove the value and the UX model, then evaluate what belongs in the composites. Copilot in Rooms is applying that same validation loop to a harder problem — in-call AI presence is a more disruptive UX change than a post-call screen, and the risk of getting the API shape wrong is higher. The Sample Builder is exactly the right surface for questions you're still figuring out.

04 Spec-Driven
Exploration

Exploratory doesn't mean unscoped

Copilot in Rooms was specced using the full UI Library feature spec template: customer-facing language, use cases, in-scope and out-of-scope scenarios, competitive landscape, privacy and regionality considerations, CELA review, functional requirements broken into phases, dependencies, compatibility requirements, risks, and open questions. The exploratory nature of the feature made the spec more important, not less — if you don't define the boundaries precisely, engineering builds the wrong thing and the iteration happens at the wrong cost.

The spec defined the Copilot side pane as supporting Rooms-based calls specifically (scoped away from Teams interop in the initial phase), Copilot Studio as the AI integration point, and a three-phase delivery model: Sample Builder experimentation, UI Library composite support, then full production guidance. Each phase had its own success criteria.

05 Strategic
Framing

What this work was really trying to prove

The ACS AI strategy was organized around three pillars: product truth (real integrations with Copilot Studio and Azure AI), customer stories (case studies that give ACS credibility in AI scenarios), and "5-minutes-to-wow" (no-code Sample Builder experiences that let anyone deploy an AI-enabled call). Copilot in Rooms was designed to serve all three — but its most important job was product truth.

The ACS envisioning work for FY25 H2 listed in-call AI under "Real-Time Assistance" alongside longer-range capabilities: voice authentication, AI interpreter, sign language recognition. Copilot in Rooms was the first, most concrete version of that category — designed to answer the interaction design question before the more complex real-time AI scenarios were committed to roadmap. What does it feel like to have an AI participant in a call? How does it enter, exit, speak, and respond? The side pane model was the beginning of an answer.

Results

Side pane

Established the spatial model for AI presence in a calling interface — a pattern that can extend to future AI features beyond Copilot (interpreter, voice authentication, real-time coaching).

Full spec

Complete feature spec with PM/EM ownership, phased delivery model, success criteria per phase, and explicit in-scope/out-of-scope definitions — exploratory rigor, not just exploratory ideation.

Sample Builder

Validated the Sample Builder's strategic role as a design validation surface — a place to test AI UX patterns before committing them to the production component API.

Product truth

The first ACS product to answer the question "what does in-call AI presence feel like?" — creating a concrete reference point for the full Real-Time Assistance AI category.

Platform shift

Completed the repositioning of the Sample Builder from a demo tool for existing features to a forward-looking surface for AI-first communication scenarios.
